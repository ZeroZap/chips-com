# UART Deep Dive

## 目标

本文深入 UART 工程细节，重点解决“能跑”和“稳定量产”之间的问题：波特率误差、采样容差、硬件流控、DMA 接收、协议分帧、低功耗唤醒和现场排障。

## UART 接收为什么能不同步

UART 没有独立时钟线。接收端依靠起始位同步：

```text
空闲高电平 -> 检测下降沿 -> 认为起始位开始 -> 按本地波特率采样后续 bit
```

这意味着 UART 的可靠性依赖：

- 双方波特率足够接近。
- 起始位边沿清晰。
- 一帧内累计时钟误差不能太大。
- 接收端采样点落在每个 bit 的稳定区域。

## 过采样

很多 UART 控制器使用过采样，例如：

```text
16x oversampling
8x oversampling
```

以 16x 为例，接收端在一个 bit 时间内有 16 个采样节拍，可以更稳地定位 bit 中心。

常见做法：

```text
检测起始位下降沿
等待半个 bit 时间
在 bit 中间附近采样
后续每隔一个 bit 时间采样一次
```

过采样越高，抗抖动能力通常越好，但对时钟和硬件设计也有具体限制。

## 波特率误差

UART 双方的实际波特率不可能完全一致。

误差来源：

- 晶振精度。
- 内部 RC 振荡器误差。
- 分频寄存器不能精确整除。
- 温度和电压漂移。
- USB 转串口工具实际误差。

常见经验：

```text
总误差最好控制在 2% 以内
高速或长帧时应更严格
内部 RC 时钟不校准时不适合高可靠高速 UART
```

如果出现稳定乱码，优先检查：

```text
系统时钟配置
UART 分频计算
实际波特率
对端参数
```

## 帧越长，误差越敏感

UART 每帧只在起始位重新同步。

例如 `8N1` 一帧是：

```text
1 start + 8 data + 1 stop = 10 bit
```

如果使用：

```text
8E2
```

帧更长，累计误差更大。

长帧配置、低质量时钟、高波特率组合在一起时，更容易出现边界采样错误。

## 校验位能解决什么

UART parity 只能检测部分单 bit 错误。

常见配置：

```text
None：无校验
Even：偶校验
Odd：奇校验
```

它不能解决：

- 丢字节。
- 字节顺序错。
- 多 bit 错误刚好校验通过。
- 帧边界混乱。
- 应用层数据完整性。

工业协议通常还会加：

```text
CRC16
Checksum
Length
Sequence
Timeout
```

## 硬件流控 RTS/CTS

高速 UART 或接收端处理能力不稳定时，可以使用硬件流控。

常见信号：

```text
RTS：Request To Send
CTS：Clear To Send
```

用途：

```text
接收端快满时通知发送端暂停
接收端有空间时允许继续发送
```

适合：

- 大量连续数据。
- 模块 AT 通信。
- 蓝牙/Wi-Fi/蜂窝模块。
- 操作系统调度不可控的场景。

如果没有流控，必须保证：

```text
接收中断/DMA 足够快
缓冲区足够大
协议允许重传或限速
```

## DMA 接收

高吞吐 UART 推荐 DMA 接收。

常见模式：

```text
DMA circular buffer
UART IDLE interrupt
ring buffer
```

典型流程：

```text
DMA 持续把 UART 数据写入环形缓冲区
IDLE 中断表示一段数据暂时结束
软件根据 DMA 写指针取出新增数据
协议解析器按状态机处理帧
```

注意点：

- IDLE 不等于完整协议帧，只表示线路空闲。
- DMA 缓冲区不能太小。
- 读写指针要处理回绕。
- Cache 系统要处理缓存一致性。
- 错误中断要清除 ORE/FE/NE 等标志。

## 常见 UART 错误标志

不同 MCU 命名不同，但常见有：

```text
ORE：Overrun Error，接收溢出
FE：Framing Error，停止位错误
PE：Parity Error，校验错误
NE：Noise Error，噪声错误
```

排查方向：

| 错误 | 可能原因 |
| --- | --- |
| ORE | 中断太慢、DMA 未开、缓冲区满 |
| FE | 波特率错、电平异常、停止位不匹配 |
| PE | 校验位配置不一致或数据损坏 |
| NE | 干扰、线缆过长、边沿质量差 |

## 协议分帧设计

UART 是字节流，不知道哪里是一条消息。

可靠协议建议包含：

```text
SOF：起始标记
LEN：长度
TYPE/CMD：命令或类型
SEQ：序号，可选
PAYLOAD：数据
CRC：校验
```

示例：

```text
AA 55 | LEN | CMD | SEQ | PAYLOAD | CRC16
```

解析器应支持：

- 从任意字节流中重新同步。
- 长度异常丢弃。
- CRC 错误丢弃。
- 超时重置状态机。
- 序号去重或重传。

## 文本协议和二进制协议

文本协议优点：

- 人能直接看。
- 串口工具方便调试。
- 适合命令行和 AT 指令。

文本协议缺点：

- 开销大。
- 解析复杂度可能被低估。
- 二进制数据需要转义或编码。

二进制协议优点：

- 高效。
- 长度明确。
- 适合嵌入式设备间通信。

二进制协议缺点：

- 人眼不易调试。
- 需要协议解析工具。
- 要处理转义、同步和 CRC。

## 低功耗唤醒

UART 可用于低功耗唤醒，但要注意：

- 唤醒后时钟是否稳定。
- 第一个字节是否会丢。
- 是否需要前导字节。
- RX 引脚电平是否能触发唤醒。
- 波特率时钟源在低功耗模式下是否可用。

常见策略：

```text
发送端先发 wakeup byte
等待目标唤醒
再发送正式协议帧
```

## 量产测试建议

UART 量产测试可以覆盖：

1. TX/RX 回环测试。
2. 标准波特率通信测试。
3. 高低温下通信稳定性。
4. 长时间连续收发压力测试。
5. 错误帧和 CRC 检测测试。
6. 断线重连测试。
7. RS485 方向控制测试。
8. 模块流控测试。

## 深度检查清单

- 实际波特率误差已测量或计算。
- 时钟源精度满足目标波特率。
- 接收使用中断或 DMA，不依赖长时间轮询。
- 错误标志有统计和清除逻辑。
- 协议有长度、校验、超时和重同步。
- 高吞吐场景已评估 RTS/CTS 或限速。
- 低功耗唤醒不会丢正式帧。
- 现场日志能区分物理层、UART 层和协议层问题。
