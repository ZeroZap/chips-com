# I3C Topic Guide

## 目标

本文面向已经理解 I2C 的读者，重点梳理 I3C 的工程价值、迁移边界、动态地址、CCC、IBI 和混挂 I2C 的注意事项。

## 适合引入 I3C 的场景

I3C 适合：

- 多传感器板内互联。
- 希望减少 GPIO 中断线。
- I2C 地址冲突明显。
- I2C 速度或功耗不能满足需求。
- 新 SoC 和目标器件都已支持 I3C。

I3C 不适合：

- 长距离外接线缆。
- 强干扰工业现场。
- 工具链和驱动尚不成熟的快速项目。
- 需要兼容大量旧 I2C 设备的系统。

## I2C 到 I3C 的思维变化

I2C 常见模式：

```text
固定地址
Controller 主动轮询
额外 GPIO 中断
开漏上拉限制速度
```

I3C 引入：

```text
动态地址分配
CCC 管理命令
IBI 带内中断
Hot-Join
更多推挽阶段
```

所以 I3C 不是简单把 I2C 频率调高。

## 动态地址分配

典型流程：

```text
Controller 初始化总线
广播 ENTDAA
Target 上报身份信息
Controller 分配 Dynamic Address
后续使用动态地址通信
```

软件设计要从“写死地址”改成：

```text
发现设备
识别设备能力
分配和记录动态地址
按运行时地址通信
```

## CCC

`CCC` 是 Common Command Code，用于总线管理。

常见用途：

- 启用或禁用事件。
- 控制 IBI。
- 分配动态地址。
- 获取设备能力。
- 管理 Hot-Join。
- 复位动态地址。

工程上要把 CCC 当成 I3C 控制面的核心。

## IBI

`IBI` 是 In-Band Interrupt。

传统 I2C：

```text
Sensor INT GPIO -> Controller GPIO
```

I3C：

```text
Sensor 通过 I3C 总线发起 IBI
Controller 收到事件后读取数据
```

IBI 适合：

- 传感器数据就绪。
- 故障事件。
- 阈值触发。
- 减少中断引脚。

## 混挂 I2C 设备

混合总线要确认：

- Controller 是否支持 mixed bus。
- I2C 设备是否会误解 I3C 时序。
- I2C 设备地址是否冲突。
- 总线速度是否因 I2C 设备被限制。
- 上拉和电容是否满足 I3C 目标速率。

如果项目目标是充分使用 I3C 特性，优先使用纯 I3C 设备链路。

## 调试步骤

1. 先确认 Controller 和 Target 都支持 I3C。
2. 低速或兼容模式确认目标存在。
3. 执行动态地址分配。
4. 读取目标能力信息。
5. 启用必要 CCC 功能。
6. 启用并验证 IBI。
7. 再提高速率或启用高级模式。

## 常见问题

| 现象 | 优先检查 |
| --- | --- |
| DAA 失败 | 设备支持、上电时序、混挂设备 |
| IBI 不触发 | CCC 配置、目标使能、控制器支持 |
| 混挂后变慢 | I2C 设备限制、上拉、电容 |
| 设备地址变化 | 动态地址生命周期、重新枚举 |
| 工具无法解码 | 分析仪是否支持 I3C |

## 实战检查清单

- 已确认是否为纯 I3C 或混合总线。
- 动态地址分配流程已实现。
- CCC 管理流程已实现。
- IBI 事件处理已实现。
- 传统 I2C 设备不会限制关键能力。
- 调试工具支持 I3C 解码。
