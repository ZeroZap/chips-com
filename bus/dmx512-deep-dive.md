# DMX512 Deep Dive

## 目标

本文深入 DMX512 的帧时序、物理层、灯具通道模型、刷新率、RDM、控制器实现和现场排障，适用于舞台灯光、建筑灯光、演出设备、灯光控制台、DMX 解码器和嵌入式灯控网关设计。

本文不重复基础通道调试，而是回答这些问题：

- Break、MAB、Start Code 和 Channel Data 如何组成完整帧。
- 为什么 DMX512 更像周期广播控制流，而不是请求响应协议。
- 灯具地址、通道模式和通道占用如何决定控制结果。
- 为什么长线、多灯和分支会导致闪烁或错色。
- RDM 为什么需要双向通信和方向控制。
- 控制器和解码器实现时如何处理丢帧、保持、刷新率和安全输出。

## 分层模型

DMX512 可按以下层次理解：

```text
Physical Layer：差分线、终端、线缆、连接器、隔离
        |
Serial Timing：250 kbps、8N2、Break、MAB
        |
DMX Frame：Start Code、Channel 1..512
        |
Fixture Mapping：起始地址、通道模式、属性映射
        |
Lighting Application：亮度、颜色、运动、效果、安全状态
```

关键点：

- DMX512 以控制器周期广播为主。
- 灯具通常只读取自己地址范围内的通道。
- 普通 DMX 没有逐灯应答。
- 现场问题常来自时序、线缆、地址和灯具模式。

## 物理层

DMX512 常使用类似 RS485 的差分物理层，但工程规范和应用场景与工业 RS485 不完全相同。

典型连接：

```text
Controller -> DMX OUT -> Fixture 1 -> Fixture 2 -> ... -> Terminator
```

物理层关注：

- DMX+ / DMX- 极性。
- 屏蔽双绞线。
- 终端电阻。
- 连接器质量。
- 隔离和地环路。
- 分支和星型拓扑。

推荐：

- 使用符合 DMX 场景的线缆。
- 长链路末端加终端。
- 避免无源 Y 分支。
- 多分支使用 DMX Splitter。
- 控制器和灯具之间考虑隔离，尤其是大型舞台和户外环境。

## 串口参数

DMX512 典型数据格式：

```text
250 kbps
8 data bits
No parity
2 stop bits
```

帧中每个数据槽通常 8-bit。

注意：

- Break 不是普通 UART 字节。
- Break 是长时间低电平，用于标记新帧开始。
- 普通 UART 外设可能需要特殊方式生成 Break。
- 接收端要能识别 Break，而不是只按字节流解析。

## 帧结构

典型 DMX 帧：

```text
Break
Mark After Break
Start Code
Channel 1
Channel 2
...
Channel N
```

最多常见 512 个通道。

Start Code：

- `0x00`：普通灯光控制数据。
- 非零 Start Code：可用于扩展用途，如 RDM 或其他特定数据。

普通灯具通常只处理 `0x00` Start Code 的帧。

## Break 和 MAB

Break 用于让接收端同步到新帧。

MAB 是 Mark After Break，Break 后保持空闲高电平的一小段时间。

调试要点：

- Break 过短，灯具可能无法识别新帧。
- MAB 过短，某些灯具可能不稳定。
- Break 和 MAB 抖动过大，可能导致部分灯具偶发丢帧。
- 控制器在不同负载或中断压力下仍要稳定生成 Break。

示波器观察顺序：

1. 看 Break 低电平宽度。
2. 看 MAB 高电平宽度。
3. 看 Start Code 是否为预期值。
4. 看通道字节是否连续。
5. 看帧间隔和刷新周期是否稳定。

## 通道和灯具模式

灯具通过起始地址和通道模式决定读取哪些数据。

示例：

```text
灯具起始地址 = 101
模式 = RGBW 4 通道

101：R
102：G
103：B
104：W
```

摇头灯可能占用更多通道：

```text
Pan coarse
Pan fine
Tilt coarse
Tilt fine
Dimmer
Strobe
Color Wheel
Gobo
...
```

常见错误：

- 起始地址错。
- 灯具模式不一致。
- 控制台 fixture profile 与灯具实际模式不一致。
- 通道重叠。
- 16-bit 粗细通道顺序理解错误。

## 刷新率和帧长度

DMX 控制器可以发送少于 512 个通道。

影响刷新率的因素：

- Break 和 MAB 时长。
- 通道数量。
- 帧间空闲时间。
- 控制器处理能力。

工程直觉：

- 通道越多，完整帧时间越长。
- 刷新率过低，动态效果可能不顺滑。
- 刷新率过高或时序不规范，某些灯具可能不兼容。
- 多 universe 系统要考虑每个 universe 的刷新和同步。

灯光场景常见要求不是微秒级控制，而是稳定、无闪烁、无错色和视觉连续性。

## 控制器实现

控制器需要稳定产生：

- Break。
- MAB。
- Start Code。
- 通道数据。
- 帧周期。

实现方式：

- UART + GPIO 控制 TX 线生成 Break。
- UART 支持 Break 发送功能。
- 定时器 + DMA 精确发送数据。
- FPGA 或专用控制芯片。

嵌入式实现注意：

- 发送缓冲区双缓冲，避免更新过程中撕裂。
- DMA 发送完成后再开始下一帧。
- 中断优先级不能影响 Break/MAB 时序。
- 多 universe 输出要避免 CPU 抖动造成不同步。
- 通道值更新和发送线程之间要有一致性策略。

## 接收器实现

灯具或解码器接收端要处理：

- Break 检测。
- Start Code 过滤。
- 通道计数。
- 起始地址匹配。
- 通道数据缓存。
- 丢帧和超时。
- 输出保持或安全状态。

典型策略：

```text
检测 Break
清通道计数
接收 Start Code
若 Start Code == 0x00，接收通道数据
到达本设备地址范围后复制数据
帧结束后更新输出
若长时间无有效帧，进入保持或安全状态
```

输出策略要明确：

- 信号丢失时保持最后值。
- 信号丢失时渐灭。
- 信号丢失时关闭输出。
- 信号恢复后是否平滑过渡。

## RDM

RDM 是 Remote Device Management，用于在 DMX 线上进行设备发现、配置和状态读取。

RDM 与普通 DMX 的关键差异：

- RDM 是双向通信。
- 控制器和设备需要方向控制。
- 需要避免多个设备同时响应。
- 使用不同 Start Code 和命令结构。

RDM 常见用途：

- 发现设备。
- 设置地址。
- 读取设备信息。
- 读取传感器或状态。
- 修改设备参数。

工程风险：

- 普通单向 DMX 分配器不支持 RDM 返回路径。
- 方向控制时序不当导致响应丢失。
- 多设备响应冲突。
- 旧灯具或线缆环境对 RDM 兼容性差。

如果系统只需要灯光控制，不一定需要启用 RDM；如果需要远程编址和状态读取，应确保链路中所有设备支持 RDM。

## 多 Universe

大型灯光系统常使用多个 universe。

每个 universe 通常最多 512 通道。

多 universe 关注：

- 通道规划。
- 控制器输出数量。
- Art-Net / sACN 到 DMX 网关映射。
- universe 编号一致性。
- 多输出同步。
- 故障隔离。

常见问题：

- 控制台 universe 编号和网关编号不一致。
- 网关输出口映射错。
- 某一 universe 刷新率异常。
- 网络到 DMX 网关延迟或丢包。

## 抗干扰和长线

舞台和建筑现场常见干扰：

- 调光器大电流。
- 电机和升降设备。
- 长距离线缆。
- 临时接线和接插件磨损。
- 地电位差。
- 户外雷击浪涌。

建议：

- 使用隔离 DMX 输出。
- 使用 DMX Splitter 分段。
- 长线末端加终端。
- 避免 DMX 线和强电长距离并行。
- 检查连接器焊点和屏蔽。
- 户外系统加浪涌和接地保护。

闪烁、随机跳色、偶发失控通常优先看线缆、终端、分支、干扰和接地。

## 现场排查流程

建议顺序：

1. 单灯、短线、近距离验证控制器输出。
2. 示波器确认 Break、MAB、Start Code 和通道数据。
3. 确认灯具起始地址和通道模式。
4. 逐步增加灯具数量。
5. 检查末端终端。
6. 加入原现场长线。
7. 加入分配器或 Splitter。
8. 检查强电、调光器和电机干扰。
9. 对多 universe 检查网关映射。
10. 对 RDM 检查双向链路和方向控制。

## 常见故障定位

| 现象 | 优先检查 |
| --- | --- |
| 灯完全不响应 | DMX+/DMX-、收发器、Break、Start Code、地址 |
| 只有第一盏正常 | 后续出线、连接器方向、终端误开、线缆断点 |
| 颜色或动作错 | 通道模式、fixture profile、起始地址、通道重叠 |
| 偶发闪烁 | 终端、线缆、干扰、刷新率、接插件松动 |
| 多灯随机跳变 | 分支、屏蔽接地、总线过长、Splitter 质量 |
| RDM 找不到设备 | 返回路径、方向控制、分配器不支持 RDM、设备兼容 |
| 网络转 DMX 异常 | Universe 映射、网关输出口、Art-Net/sACN 配置 |

## 深度检查清单

- 控制器输出 250 kbps、8N2 正确。
- Break 和 MAB 时序稳定。
- Start Code 符合应用预期。
- 通道数量和刷新率满足场景需求。
- 灯具起始地址和通道模式一致。
- fixture profile 与灯具手册一致。
- 长线末端终端正确。
- 分支使用合适 Splitter。
- 丢帧后输出保持/渐灭/关闭策略明确。
- RDM 场景确认双向链路和方向控制。
- 多 universe 编号、网关输出口和控制台配置一致。
- 户外或大型舞台有隔离、浪涌和接地策略。

## 与 Modbus RTU 的区别

DMX512 和 Modbus RTU 都可能使用类似 RS485 的差分物理层，但通信模型完全不同。

DMX512：

```text
控制器周期广播
灯具按地址读取通道
普通 DMX 无逐灯应答
强调视觉连续性和低故障感知
```

Modbus RTU：

```text
主站请求
从站响应
寄存器读写
强调数据正确性和异常响应
```

不要用 Modbus 的请求响应思路理解普通 DMX，也不要用 DMX 的广播思路设计工业寄存器控制。
