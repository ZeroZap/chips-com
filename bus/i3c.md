# I3C

## 定位

I3C 是 MIPI 联盟推出的芯片间串行总线，用于改进传统 I2C 在速度、功耗、地址管理和中断机制上的限制。它兼容部分 I2C 设备生态，常用于传感器、低速外设、板内管理和移动设备内部互联。

I3C 可以理解为 I2C 的演进方向之一：保留两线连接方式，但引入更高的数据速率、动态地址分配、带内中断和更完善的总线管理能力。

## 基本概念

- `SCL`：串行时钟线，由当前控制器驱动。
- `SDA`：串行数据线，承载地址、数据、控制信息和带内中断。
- `Controller`：总线控制器，类似 I2C 中的主机角色。
- `Target`：总线目标设备，类似 I2C 中的从机角色。
- `Dynamic Address`：动态地址，由控制器在总线初始化或设备加入时分配。
- `CCC`：Common Command Code，I3C 定义的通用控制命令。
- `IBI`：In-Band Interrupt，目标设备通过总线数据线主动发起的带内中断。

## 与 I2C 的关系

I3C 继承了 I2C 的双线结构，但不是简单的高速 I2C。它在协议层面增加了更多总线管理能力。

| 项目 | I2C | I3C |
| --- | --- | --- |
| 信号线 | SCL + SDA | SCL + SDA |
| 地址方式 | 静态地址为主 | 动态地址为主 |
| 中断方式 | 额外 GPIO 中断线常见 | 支持带内中断 IBI |
| 速度 | 常见 100 kHz / 400 kHz / 1 MHz | SDR 模式可达更高速率 |
| 多设备管理 | 地址冲突需要硬件或软件规避 | 支持动态地址分配 |
| 兼容性 | I2C 设备之间兼容性较好 | 可在一定条件下兼容 I2C 目标设备 |

## 硬件连接

I3C 仍然使用 `SCL` 和 `SDA` 两根信号线，因此在硬件形态上接近 I2C。但设计时需要注意：

- 总线上的 I2C 兼容设备可能限制 I3C 的速度和功能。
- 上拉电阻、电容负载、走线长度会影响最高通信速率。
- 混挂 I2C 设备时，需要确认控制器是否支持对应兼容模式。
- 如果系统依赖传统外部中断线，迁移到 I3C 后可以评估是否改用 IBI。

## 通信流程

典型 I3C 总线流程包括：

1. 控制器初始化总线。
2. 发现总线上的目标设备。
3. 通过动态地址分配为设备分配地址。
4. 使用普通读写或 CCC 命令进行通信。
5. 目标设备可通过 IBI 主动通知控制器事件。
6. 控制器根据设备能力选择合适的数据速率和通信模式。

## 优点与限制

优点：

- 比传统 I2C 速度更高。
- 动态地址减少地址冲突问题。
- IBI 可以减少额外中断引脚。
- 更适合多个传感器和低速管理设备挂在同一总线上。
- 在新平台上有逐步普及趋势。

限制：

- 生态成熟度仍低于 I2C、SPI、UART。
- 控制器和目标设备都需要明确支持 I3C。
- 混挂 I2C 设备时，并非所有 I3C 特性都能使用。
- 调试工具、驱动支持和工程经验相对少一些。

## 典型应用

- 手机、平板、笔记本内部传感器互联。
- IMU、加速度计、陀螺仪、磁力计等传感器总线。
- 电源管理、温度监控、系统管理设备。
- 需要减少中断引脚数量的多外设系统。
- 对 I2C 速度和地址管理能力不满意，但又希望保留双线结构的设计。

## 与相关协议对比

- 相比 `I2C`：I3C 更快，支持动态地址和带内中断，但生态和调试便利性不如 I2C 成熟。
- 相比 `SPI`：I3C 引脚更少，更适合多设备共享总线；SPI 通常吞吐更高、实现更简单。
- 相比 `UART`：I3C 更适合板内多设备同步通信；UART 更适合点对点异步通信或通过收发器扩展距离。
- 相比 `SMBus/PMBus`：I3C 更偏底层总线演进，SMBus/PMBus 更偏系统管理和电源管理协议规范。

## 学习建议

学习 I3C 前建议先理解：

1. `I2C` 的开漏、上拉、地址、ACK/NACK 和多设备挂载。
2. `SPI` 的同步通信和高速外设连接方式。
3. 中断引脚在传统传感器系统中的作用。
4. 动态地址和总线枚举的意义。
