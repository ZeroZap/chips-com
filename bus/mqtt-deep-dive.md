# MQTT Deep Dive

## 目标

本文深入 MQTT 嵌入式可靠性设计，重点覆盖会话、QoS、保留消息、遗嘱、离线缓存、TLS、Topic 权限、网关映射和云端协同。

## Session

MQTT 会话影响订阅和离线消息处理。

要明确：

- Clean Session / Clean Start。
- Session Expiry。
- 断线后是否保留订阅。
- 离线消息是否缓存。

## QoS 语义

QoS 0：

```text
最多一次，不确认
```

QoS 1：

```text
至少一次，可能重复
```

QoS 2：

```text
只有一次，状态机复杂
```

业务层要根据 QoS 处理重复和丢失。

## Retained Message

保留消息适合保存最新状态，例如：

```text
设备在线状态
当前配置版本
最后遥测摘要
```

不适合滥用为历史数据存储。

## Last Will

Last Will 用于设备异常断线时由 Broker 发布遗嘱消息。

常用于：

```text
设备离线通知
网关掉线告警
```

## 离线缓存

设备断网时要决定：

- 哪些数据缓存。
- 缓存多长时间。
- 缓存上限。
- 恢复后是否补发。
- 补发是否保序。

遥测数据和告警数据策略通常不同。

## TLS 和证书

TLS 关注：

- 根证书。
- 客户端证书。
- 私钥保护。
- 时间同步。
- 证书轮换。
- RAM 占用。

设备时间不准会导致证书校验失败。

## Topic 权限

Topic 设计要配合 ACL。

原则：

- 设备只能发布自己的遥测。
- 设备只能订阅自己的命令。
- 网关要隔离子设备权限。
- 避免通配符权限过宽。

## 网关映射

工业网关常见：

```text
Modbus/CAN/RS485 -> MQTT
```

映射要定义：

- 采集周期。
- 数据单位。
- 时间戳来源。
- 断线缓存。
- 命令下发确认。
- 子设备在线状态。

## 检查清单

- Client ID 唯一且稳定。
- Session 策略明确。
- QoS 与业务匹配。
- 重复消息可处理。
- Last Will 配置正确。
- TLS 时间和证书管理正确。
- Topic ACL 不过宽。
- 离线缓存有上限和策略。
